## 3. Descripción del Sistema

### 3.1 Propósito

BioCore Medical es un sistema integral de gestión hospitalaria desarrollado para digitalizar y automatizar los procesos clínicos y administrativos de un hospital. Cubre desde la recepción del paciente hasta la entrega de resultados de laboratorio, despacho de medicamentos y gestión de pagos.

### 3.2 Usuarios del sistema

| Rol | Descripción | Accesos principales |
|-----|-------------|---------------------|
| `ADMIN` | Administrador del sistema | Todo + reportería + gestión de personal |
| `HEALTH_STAFF` | Personal de salud / Recepcionista / Enfermería | Recepción, cola, signos vitales, emergencias |
| `DOCTOR` | Médico (clínica normal) | Consulta médica, prescripciones, órdenes de lab |
| `DOCTOR` (emergencia) | Médico de emergencias | Emergencias médicas, reportes clínicos |
| `LAB_TECHNICIAN` | Laboratorista | Órdenes de laboratorio, resultados |
| `PHARMACIST` | Farmacéutico | Inventario, ventas, despacho de recetas |
| `CASHIER` | Cajero | Pagos, citas, facturación |
| `PATIENT` | Paciente | Portal de citas y resultados propios |

### 3.3 Reglas de negocio principales

| Código | Regla |
|--------|-------|
| RN-P003 | Usuario con contraseña temporal debe cambiarla en el primer inicio de sesión |
| RN-M01 | Cada clínica tiene un máximo de doctores asignados |
| RN-M02 | Un doctor solo puede estar activo en una clínica a la vez |
| RN-03 | Signos vitales son obligatorios antes de pasar al médico |
| RN-C01 | Pacientes URGENTES van al frente de la cola |
| RN-L01 | Órdenes de laboratorio tienen validez de 30 días |
| RN-L02 | Notificación por correo cuando resultados de lab están disponibles |
| RN-F01 | Stock de medicamentos debe actualizarse en cada venta |
| RN-F02 | Medicamento de prescripción solo se despacha tras pago confirmado |
| RN-P01 | Descuento por seguro se aplica automáticamente al calcular pago |
| RN-P02 | `netAmount = amount - discountAmount` |
| RN-R03 | Dashboard se actualiza cada 10 segundos en tiempo real |
| RN-05 | Documentos adjuntos solo en formato PDF, máximo 20 MB |

## 4. Arquitectura del Sistema

### 4.1 Estilo arquitectónico

BioCore Medical sigue una **arquitectura cliente-servidor de 3 capas** desacoplada, con comunicación exclusivamente a través de una API REST con autenticación JWT stateless.

```
┌──────────────────────────────────────────────────┐
│                CLIENTE (SPA)                     │
│         Angular 17 — biocore-frontend            │
│   Componentes · Servicios · Guards · Interceptors│
└─────────────────────┬────────────────────────────┘
                      │  HTTPS · REST · Bearer JWT
┌─────────────────────▼────────────────────────────┐
│              SERVIDOR DE APLICACIÓN              │
│       Spring Boot 3.3.6 — biocore-backend        │
│  Controllers · Services · Security · JPA/ORM     │
└─────────────────────┬────────────────────────────┘
                      │  JDBC · SSL
┌─────────────────────▼────────────────────────────┐
│             BASE DE DATOS                        │
│         PostgreSQL — Neon Cloud                  │
│         23 tablas · pgBouncer pooling            │
└──────────────────────────────────────────────────┘
```

### 4.2 Patrón de comunicación

- El frontend **nunca accede directamente** a la base de datos.
- Toda operación pasa por la API REST del backend (`/api/**`).
- Las rutas públicas (`/api/auth/**`, `/api/public/**`) no requieren token.
- Todas las demás rutas requieren un **Bearer JWT** inyectado automáticamente por `AuthInterceptor`.
- El backend valida el token en cada request mediante `JwtFilter` (filtro `OncePerRequestFilter`).

### 4.3 Modelo de seguridad

```
Request HTTP
    │
    ▼
JwtFilter (OncePerRequestFilter)
    │  extrae "Authorization: Bearer <token>"
    │  valida firma HMAC-SHA con JwtUtil
    │  carga UserDetails y setea SecurityContext
    ▼
SecurityFilterChain
    │  verifica permisos por ruta y rol
    ▼
Controller → Service → Repository → PostgreSQL
```

- Algoritmo de firma: **HMAC-SHA** (clave configurable vía env var `JWT_SECRET`)
- Expiración del token: **24 horas** (`APP_JWT_EXPIRATION=86400000 ms`)
- Gestión de sesiones: **STATELESS** (sin HttpSession en servidor)
- Contraseñas: cifradas con **BCrypt**
- CORS: restringido a los orígenes configurados en `APP_CORS_ALLOWED_ORIGINS`

## 5. Capas de la Aplicación

### 5.1 Capas del Backend (Spring Boot)

```
com.biocore/
├── config/          → Capa de Configuración
├── controller/      → Capa de Presentación (REST API)
├── service/         → Capa de Lógica de Negocio
├── repository/      → Capa de Acceso a Datos
├── entity/          → Capa de Dominio (Entidades JPA)
├── dto/             → Objetos de Transferencia de Datos
├── enums/           → Enumeraciones de dominio
└── security/        → Capa de Seguridad transversal
```

#### Descripción de cada capa

| Capa | Paquete | Responsabilidad |
|------|---------|-----------------|
| **Configuración** | `config/` | `SecurityConfig` (CORS, filtros, BCrypt), `DataInitializer` (seed), `SchemaMigrationRunner` |
| **Presentación** | `controller/` | 20 `@RestController` exponen endpoints REST. Reciben requests HTTP, delegan a services, retornan `ApiResponse<T>` |
| **Lógica de Negocio** | `service/` | 16 clases `@Service` con las reglas de negocio (RN-*). Orquestan entidades, repositorios y servicios externos |
| **Acceso a Datos** | `repository/` | Interfaces `JpaRepository` + queries JPQL personalizadas. Sin SQL manual |
| **Dominio** | `entity/` | 23 entidades `@Entity` mapeadas a tablas PostgreSQL vía Hibernate |
| **DTOs** | `dto/` | Request/Response objects para desacoplar API del modelo interno |
| **Seguridad** | `security/` | `JwtUtil`, `JwtFilter`, `UserDetailsServiceImpl`, `CustomUserDetails` |

#### Servicios de la capa de negocio

| Servicio | Responsabilidad principal |
|----------|--------------------------|
| `AuthService` | Login, generación de JWT, cambio de contraseña |
| `UserService` | CRUD de usuarios, asignación a clínicas, disponibilidad del doctor |
| `PatientService` | Registro, búsqueda por DPI, actualización de expedientes |
| `TicketService` | Gestión de cola: crear, llamar, signos vitales, completar, ausentar |
| `AppointmentService` | Agendación, slots disponibles, reprogramación |
| `VitalSignsService` | Registro de signos vitales por ticket |
| `LabService` | Órdenes de laboratorio, recolección, carga de resultados PDF |
| `PrescriptionService` | Creación y despacho de recetas médicas |
| `PharmacySaleService` | Venta OTC, despacho con receta, actualización de stock |
| `PaymentService` | Creación y procesamiento de pagos, descuentos por seguro |
| `EmergencyService` | Registro de emergencias, reporte médico, integración pacientes |
| `ReportService` | Dashboard tiempo real, reportes administrativos |
| `EmailService` | Notificaciones vía SendGrid HTTP API (RN-L02) |
| `DocumentService` | Subida y validación de PDFs con Apache PDFBox |
| `DoctorScheduleService` | Horarios de médicos por clínica |
| `ClinicScheduleService` | Horarios de clínicas |

### 5.2 Capas del Frontend (Angular 17)

```
src/app/
├── core/            → Capa de Núcleo (singleton, carga una sola vez)
│   ├── auth/        → AuthService (Signals + JWT)
│   ├── guards/      → authGuard (CanActivateFn)
│   ├── interceptors/→ AuthInterceptor (Bearer token automático)
│   └── models/      → Interfaces TypeScript del dominio
├── shared/          → Capa Compartida (servicios reutilizables)
│   └── services/    → 14 servicios HTTP (PatientService, TicketService, etc.)
└── modules/         → Capa de Presentación (15 módulos funcionales)
    └── [módulo]/    → Componentes standalone con lazy loading
```

#### Descripción de cada capa

| Capa | Responsabilidad |
|------|-----------------|
| **Core** | Servicios singleton: autenticación con Angular Signals, guard de rutas, interceptor HTTP que inyecta el Bearer token en cada request |
| **Shared** | Servicios HTTP reutilizables. Cada servicio encapsula las llamadas REST a un dominio (patients, tickets, lab, payments…) |
| **Modules** | Componentes Angular standalone con lazy loading. Cada módulo corresponde a un rol/flujo funcional del sistema |

#### Gestión de estado reactivo

BioCore utiliza **Angular Signals** (no NgRx ni servicios con BehaviorSubject) para el estado del usuario autenticado:

```typescript
// AuthService
currentUser = signal<LoginResponse | null>(null);
doctorClinicType = signal<string | null>(null);

// AppComponent — computed para filtrar nav items por rol
visibleNavItems = computed(() => 
  this.navItems.filter(item => this.authService.hasRole(...item.roles))
);
```

## 6. Frameworks y Tecnologías Principales

### 6.1 Angular 17 — Frontend Framework

Angular 17 es el framework SPA del frontend. Las características clave utilizadas en el proyecto son:

| Característica | Uso en BioCore |
|----------------|----------------|
| **Standalone Components** | Todos los componentes son standalone (sin NgModule) |
| **Lazy Loading** | Cada ruta carga su componente bajo demanda con `loadComponent()` |
| **Angular Signals** | Estado reactivo del usuario autenticado (`currentUser`, `doctorClinicType`) |
| **Angular Material (MDC)** | Sistema de UI completo: sidenav, tablas, formularios, diálogos, snackbars |
| **Reactive Forms** | Formularios con validación en módulos de registro, consulta y pagos |
| **RxJS Observables** | Comunicación HTTP, polling del dashboard (`interval(10000)`) |
| **HttpInterceptor** | Inyección automática del Bearer token en todas las peticiones autenticadas |
| **CanActivateFn (authGuard)** | Protección de rutas; redirige a `/login` si no hay sesión activa |
| **Environment files** | `environment.ts` (dev) y `environment.prod.ts` (Railway) con `apiUrl` distinto |

### 6.2 Spring Boot 3.3.6 — Backend Framework

Spring Boot 3.3.6 sobre Java 17 es el framework del backend. Las características utilizadas:

| Característica | Uso en BioCore |
|----------------|----------------|
| **Spring Web (Tomcat embebido)** | Servidor de aplicaciones embebido, puerto configurable via `PORT` env var |
| **Spring Security 6** | Filtro JWT stateless, BCrypt, CORS configurado, `@EnableMethodSecurity` |
| **Spring Data JPA + Hibernate** | ORM completo, `ddl-auto=update`, repositorios con queries JPQL |
| **Spring Validation** | `@Valid` en DTOs de entrada para validación declarativa |
| **Spring Mail / SendGrid** | Notificaciones de resultados de laboratorio a pacientes |
| **Multipart File Upload** | Recepción de PDFs de resultados (máx 20 MB, validados con PDFBox) |
| **Lombok** | `@Data`, `@Builder`, `@RequiredArgsConstructor` para reducir boilerplate |
| **`CommandLineRunner`** | `DataInitializer` ejecuta seed de datos al arrancar la aplicación |

### 6.3 JJWT 0.12.6 — Librería JWT

Implementación completa del ciclo de vida del token:

```
Login exitoso
    │
    ▼
JwtUtil.generateToken(UserDetails)
    │  Claims: role, firstName, lastName, userId, sub, iat, exp
    │  Firma: HMAC-SHA (secret de 256 bits)
    ▼
Token devuelto al frontend → almacenado en localStorage

Cada request posterior
    │
    ▼
AuthInterceptor → Authorization: Bearer <token>
    │
    ▼
JwtFilter → JwtUtil.isTokenValid() → SecurityContext
```

### 6.4 PostgreSQL + Hibernate — Capa de Datos

| Aspecto | Configuración |
|---------|---------------|
| Dialecto | `PostgreSQLDialect` |
| DDL | `ddl-auto=update` (Hibernate gestiona el esquema automáticamente) |
| Fetch | `open-in-view=true` (sesiones abiertas durante el request completo) |
| Pool | pgBouncer vía Neon Cloud |
| SSL | Requerido (`sslmode=require`) |
| Show SQL | Desactivado en producción (`show-sql=false`) |

## 8. Características de Construcción

### 8.1 Build del Backend (Maven)

El backend utiliza **Apache Maven 3.x** con el plugin de Spring Boot para generar un JAR ejecutable.

#### Comando de build
```bash
mvn clean package
# Genera: target/biocore-backend-1.0.0.jar (fat JAR con Tomcat embebido)
```

#### Configuración del plugin (`pom.xml`)
```xml
<build>
  <plugins>
    <plugin>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-maven-plugin</artifactId>
      <configuration>
        <excludes>
          <!-- Lombok solo en compilación, no en runtime -->
          <exclude>
            <groupId>org.projectlombok</groupId>
            <artifactId>lombok</artifactId>
          </exclude>
        </excludes>
      </configuration>
    </plugin>
  </plugins>
</build>
```

#### Parent POM heredado
```xml
<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.3.6</version>
</parent>
```

El `spring-boot-starter-parent` provee:
- Versiones gestionadas de todas las dependencias Spring
- Configuración de compilación Java 17
- Plugin de Maven Surefire para tests
- Filtrado de recursos con perfiles

#### Características especiales del build backend

| Característica | Detalle |
|----------------|---------|
| **Fat JAR** | Tomcat embebido — no requiere servidor de aplicaciones externo |
| **Perfiles** | Puerto configurable via env var `PORT` (default: 8085) |
| **Lombok** | Solo en `compile scope`, excluido del JAR final |
| **DevTools** | Solo en `runtime/optional`, no incluido en producción |
| **Validación de PDF** | Apache PDFBox 3.0.3 incluido para validar archivos subidos |
| **Seed de datos** | `DataInitializer` (CommandLineRunner) ejecuta al arrancar: admin, seguros, clínicas, 45 medicamentos, 40 exámenes de laboratorio |
| **Migraciones** | `SchemaMigrationRunner` ejecuta scripts de migración de esquema al iniciar |

    participant ReportSvc as ReportService
    participant API as Backend API

    Dashboard->>RxJS: ngOnInit - interval(10000).pipe(startWith(0))

    loop Cada 10 segundos
        RxJS->>ReportSvc: getDashboard()
        ReportSvc->>API: GET /reports/dashboard
        API-->>Dashboard: totalPatientsToday, patientsWaiting...
        Dashboard->>Dashboard: Actualiza tarjetas estadísticas
    end

    Dashboard->>Dashboard: ngOnDestroy - sub.unsubscribe()
